# 地下空间智能巡检机器人系统需求文档

## 1. 项目背景

大型工业产业园的地下空间，包括地下通道、地下仓库与地下停车场，长期处于光线不足、人流车流交错、隐患不易察觉的环境中。地面裂缝坑洼、路面积水湿滑、遗落危险异物、人员违规抽烟滞留等问题，往往需要依赖人工夜间巡检才能发现。然而人工巡检不仅成本高、强度大，还存在夜间视线受限、覆盖不完整、隐患响应滞后等固有短板，一旦漏检就可能酿成安全事故。

本项目基于 ROS2 移动机器人平台，构建一套集远程控制、自主巡检、异常检测、声光提醒、自然语言辅助决策与数据分析于一体的智能巡检系统，用机器人替代人工完成重复、枯燥且存在风险的夜间巡检作业。

系统由三部分协同构成：

- ROS2 巡检小车：作为执行载体，深入地下空间一线，负责移动、建图、导航、感知、声光提醒与现场响应。
- FastAPI 后端服务：作为业务中枢，统一调度数据与指令，负责鉴权、接口网关、机器人状态聚合、告警管理、LLM 任务规划与多车协同。
- 跨平台移动端 App：作为人机交互终端，面向值班员、运维人员与管理员，提供总控看板、遥控接管、地图导航、告警处置、巡检任务配置与 LLM 助手入口。

三者共同形成“发现问题、判断风险、下发动作、现场响应、记录归档”的完整巡检闭环。

## 2. 建设目标

本系统面向地下/园区巡检场景，重点不是单独展示某个算法模块，而是打通可落地的场景能力：

1. 支持管理人员远程查看小车状态、地图、摄像头画面、告警与巡检结果。
2. 支持现场人员通过 App 对单车或多车进行低延迟遥控接管。
3. 支持 SLAM 地图展示、初始位姿设置、目标点下发与自主导航。
4. 支持巡航过程中对路面裂缝、积水湿滑、危险异物等隐患进行 AI 检测并生成告警。
5. 支持告警事件的抓拍、分类、筛选、详情查看、备注与处理闭环。
6. 支持小车在危险、避让、巡航、到达、救援等场景下触发灯光效果与语音/音频提醒。
7. 支持 LLM 任务助手将自然语言转化为受控任务计划，并在用户确认后调用安全白名单工具。
8. 支持多车在线状态聚合、批量指令、编队、狭窄通道协同、故障救援与护送返航等协同场景。
9. 支持自动化部署、运行状态检查、日志留存与算法/业务能力持续迭代。

## 3. 系统角色与权限

### 3.1 管理员

管理员拥有系统全部功能权限，负责用户账号管理、地图与航点配置、视觉模型参数配置、多车调度策略设置、系统参数维护，同时可处理全部告警并查看完整操作审计日志。管理员通常由园区信息化负责人或项目技术负责人担任。

### 3.2 值班员

值班员面向日常驻场巡检人员，能够访问总控数据看板、实时查看车辆位置与状态、处理告警事件、配置定时巡检任务，以及通过 LLM 对话模块查询巡检记录、生成任务计划或辅助生成巡检报告。值班员不能进行用户管理与底层系统配置，避免误操作影响系统稳定运行。

### 3.3 运维人员

运维人员面向现场设备维护人员，权限聚焦于单车层面：能够对指定小车进行远程遥控、查看告警详情与自动抓拍截图、在异常记录上添加处置备注，以及回放历史巡检轨迹与告警记录，便于现场核查与设备维护。

## 4. 总体架构

系统采用“移动端 App - 后端服务 - 机器人端”三层架构。

```text
移动端 App
  ├─ TCP 直连小车：低延迟遥控、灯光、音频等现场即时控制
  └─ REST 访问后端：鉴权、地图、告警、AI、LLM、多车、业务数据

FastAPI 后端服务
  ├─ web_api: 业务网关、鉴权、告警、LLM、多车、数据接口
  ├─ ros_bridge: ROS2 HTTP 桥接，发布 /cmd_vel、/goal_pose、/initialpose，读取 /map、/odom、/scan
  ├─ ai_service: ROS 图像抓帧、YOLO 检测、摄像头快照/MJPEG
  └─ MQTT 服务: 机器人心跳、状态、命令 ACK、告警推送

ROS2 巡检小车
  ├─ tcp_car_bridge: 接收 App TCP 原厂协议并转换为底盘、灯光、音频动作
  ├─ robot_agent: MQTT 心跳、状态、任务接收与 ACK
  ├─ Yahboom ROS2 底盘/雷达/导航节点
  └─ 摄像头、激光雷达、深度相机与本地 AI 推理环境
```

该架构同时满足两类需求：

- 现场调试和紧急接管需要低延迟，因此 App 通过 TCP 直连小车控制桥。
- 日常巡检业务需要鉴权、记录、告警和多车统一管理，因此地图、告警、LLM、多车调度等走后端统一接口。

## 5. 功能性需求分析

本节按照地下空间巡检的业务功能模块组织需求。每个模块先说明该模块服务的巡检场景，再拆分为若干子能力，描述系统已经具备的功能与业务效果。

### 5.1 巡检态势总览功能

巡检态势总览面向值班员和管理员，用于在夜间巡检过程中快速掌握“车辆是否在线、任务是否执行、隐患是否产生、系统是否健康”。该模块作为 App 首页或总控入口，帮助用户在不进入底层配置页面的情况下完成全局判断。

车辆在线状态监控：
系统展示每台巡检小车的在线/离线状态、当前工作模式、电量、网络延迟、最近心跳时间和最近一次状态更新时间。车辆状态由小车端 `robot_agent` 通过 MQTT 主动上报，后端进行聚合后提供给移动端展示。

巡检任务状态展示：
系统展示小车当前是否处于空闲、巡航、遥控、跟随、救援等状态，并在任务执行过程中展示当前任务名称、目标区域、执行阶段和异常中断原因，便于值班员判断巡检是否正常推进。

告警态势统计：
系统统计当前未处理告警数量、高危告警数量、不同告警类型数量以及最近告警时间，帮助值班员优先处理积水、裂缝、危险异物等高风险事件。

服务健康状态提示：
系统展示后端服务、MQTT 通信、ROS 桥接、AI 检测服务、摄像头预览与地图服务的健康状态。当 App 无法获取地图、告警或摄像头数据时，系统提示用户优先检查对应服务，而不是仅显示空白页面。

快捷业务入口：
系统从总览页提供地图导航、远程遥控、巡检任务配置、告警中心、LLM 助手、系统设置等入口，使值班员能够从态势感知快速进入处置动作。

### 5.2 远程遥控与紧急接管功能

远程遥控功能用于现场调试、狭窄区域通过、异常点位靠近和紧急接管。当地下空间出现障碍物、导航异常或需要人工观察某个隐患点时，操作人员能够直接接管小车运动。

目标小车选择：
App 允许用户选择需要控制的小车，并支持配置小车 IP、TCP 端口和后端 API 地址。多车环境下，用户能够明确区分小车 1、小车 2 等不同控制对象，避免误控。

基础运动控制：
系统提供前进、后退、左移、右移、左旋、右旋、停止、刹停等基础运动控制能力。控制指令驱动真实底盘运动，不停留在界面模拟。

摇杆连续控制：
系统支持摇杆式连续控制。用户按住摇杆时，App 周期性发送运动指令；用户松开摇杆后，系统立即发送停止指令，防止小车持续运动造成碰撞。

低延迟 TCP 控制：
现场遥控优先通过 App 直连小车 `tcp_car_bridge`，由 TCP 控制桥将原厂协议转换为底盘控制指令。该链路用于满足毫秒级响应要求，适合现场调试和紧急干预。

平台化遥控控制：
系统也支持通过后端 REST 接口下发 `/cmd_vel` 指令，由 `web_api` 转发至 `ros_bridge`，再发布到 ROS2 控制话题。该链路用于统一鉴权、日志记录和远程平台调度。

多车批量控制：
系统支持对多台选中车辆批量下发低速运动或停止指令，并为每台车返回独立执行结果。当某台车控制失败时，不影响其他车辆的结果反馈。

紧急停止控制：
系统提供一键停止能力。紧急停止具有最高优先级，可覆盖巡航、导航、灯光秀、多车协同和 LLM 生成的运动任务。

### 5.3 地图定位与自主导航功能

地图定位与自主导航功能用于解决地下空间 GPS 不可用的问题。系统已经将 ROS2 SLAM 地图、雷达点和机器人位姿接入 App，使操作人员能够在地图上理解环境并下发目标点。

SLAM 地图展示：
系统在 App 地图页展示真实 `/map` 栅格地图，包括地图尺寸、分辨率、原点信息和占用栅格数据。地图展示来自 ROS2 实时数据或保存地图，不使用静态示意图替代。

机器人位姿展示：
系统展示机器人在地图坐标系下的位置和朝向，并结合 `/odom` 或 `/amcl_pose` 更新当前位置。地图页面帮助用户判断小车当前所在区域和运动方向。

雷达点可视化：
系统将激光雷达 `/scan` 数据投影到地图坐标系，并在 App 地图上显示为雷达点，辅助用户判断前方障碍物和环境轮廓。

初始位姿校准：
系统支持用户在 App 地图上选择初始位置和朝向，并通过后端发布 `/initialpose`。该功能用于导航模式启动后校准 AMCL 位姿，避免定位漂移影响导航效果。

导航目标下发：
系统支持用户在地图上点选目标位置和朝向，并通过后端发布 `/goal_pose`。目标点由原厂 ROS2 导航节点负责路径规划和底盘运动，App 不重复实现路径规划算法。

导航状态联调：
系统能够区分“目标点已成功发布”和“小车已成功规划并运动”两个阶段。当 ROS 侧收到 `/goal_pose` 但小车不动时，优先检查导航 launch、TF、局部规划器和 `/cmd_vel` 订阅关系。

### 5.4 自主巡航与任务配置功能

自主巡航功能用于让小车按照预设路线覆盖地下通道、仓库和停车场，减少人工持续遥控。任务配置功能用于将地图上的关键点位组织成可执行巡检路线。

巡检航点管理：
系统支持创建、编辑和删除巡检航点。每个航点包含名称、地图坐标、朝向、所属区域和备注，用于描述需要巡检或停留观察的位置。

巡检路线管理：
系统支持将多个航点组合为巡检路线，并允许用户调整航点顺序。路线用于常规夜间巡检、重点区域复查和演示场景。

巡检任务启动：
用户能在 App 中选择巡检路线并启动任务。系统将路线转换为连续导航目标，驱动小车按顺序到达各航点。

巡检任务中断：
在巡检过程中，如果发生高危告警、人工接管、通信中断、紧急停止或导航失败，系统暂停或停止继续下发剩余目标点，并记录任务中断原因。

巡检状态反馈：
系统在任务执行过程中展示当前目标点、已完成航点、剩余航点、任务模式和小车状态，使值班员能够判断巡检进度。

巡检事件触发：
巡航过程中检测到裂缝、积水、异物等风险时，系统触发告警、灯光、音频、多车避让或 LLM 辅助处置流程，形成从巡航到处置的业务闭环。

### 5.5 道路与环境异常检测功能

道路与环境异常检测功能用于替代人工在弱光地下空间中识别安全隐患。系统利用摄像头图像和本地 AI 模型，在小车巡航或遥控过程中识别路面和环境风险。

摄像头图像接入：
系统通过 ROS 图像话题获取摄像头数据。当前使用 `usb_cam` 发布的 `/image_raw` 作为 RGB 图像来源，App 预览和 AI 检测复用该话题，避免多个进程同时占用物理摄像头。

实时画面预览：
App 支持查看小车摄像头快照或 MJPEG 预览，用于值班员远程确认现场情况。预览链路由 `web_api` 代理 `ai_service` 获取最新 ROS 图像帧。

单帧检测：
系统支持对单张图片或 ROS 图像话题中的一帧进行检测，返回检测类别、置信度、模型执行结果和设备信息。该能力用于调试模型和验证现场风险点。

连续巡检检测：
系统支持启动和停止连续检测任务。连续检测启动后，AI 服务按设定间隔抓取图像并运行模型，累计检测帧数、告警帧数和最近错误信息。

道面缺陷检测：
系统识别地面裂缝、坑洼等道面缺陷，发现后生成道面缺陷类告警，并保留抓拍图、检测结果和位置信息。

湿滑积水检测：
系统识别路面积水、湿滑或渗漏等风险，发现后生成积水湿滑类告警，提醒现场人员绕行或处理。

危险异物检测：
系统识别地下通道或停车场内的遗落物、障碍物、危险异物等目标，发现后生成危险异物类告警，并触发小车避让或多车绕行策略。

人员异常检测：
系统识别抽烟、滞留、跌倒、明火等人员或环境异常行为，并复用现有告警中心进行展示、筛选和处置。

模型类别管理：
AI 检测模块支持多类检测目标统一接入。新增或切换检测类别时，告警、抓拍、列表筛选和详情查看流程保持一致。

### 5.6 告警中心与闭环处置功能

告警中心用于承接 AI 检测、巡检任务和多车协同产生的异常事件。系统不仅要发现隐患，还要支持值班员完成记录、查看、备注、处理和追溯。

实时告警生成：
当系统检测到裂缝、积水、危险异物、人员异常或其他风险事件时，自动生成告警记录，包含告警编号、类型、风险等级、发生时间、小车编号、相机编号、地图坐标和图片路径。

告警分类展示：
告警中心按告警类型展示事件，包括积水湿滑、道面裂缝、危险异物、抽烟/人员异常等。用户能够快速筛选某一类风险。

风险等级筛选：
系统支持按高、中、低风险等级筛选告警，高危告警优先提示，便于值班员第一时间处理。

处理状态管理：
系统区分未处理和已处理告警。有权限用户能够将告警标记为已处理，并填写处置备注。

告警详情查看：
用户打开告警详情时，能够查看抓拍图、告警类型、风险等级、时间、小车编号、相机编号、地图位置、处理状态和备注信息。

告警图像回看：
系统提供告警图片访问接口，使 App 能展示触发告警时的现场截图，辅助运维人员判断是否需要现场处置。

操作留痕归档：
告警处理操作记录处理人、处理时间和备注内容，便于审计、复盘和统计分析。

### 5.7 声光联动与现场提醒功能

声光联动功能用于增强地下空间中的现场可感知性。不同巡检状态和风险事件会触发不同灯光与音频反馈，使现场人员能够快速判断小车状态和危险等级。

基础灯光控制：
系统支持灯光关闭、流水、跑马、呼吸、渐变、星光等基础灯光效果。灯效既能通过 App 手动触发，也能由巡检事件自动触发。

灯光秀控制：
系统支持启动和停止灯光秀，用于演示、调试或现场提示。灯光秀运行时，运动控制和紧急停止仍保持可用。

音频提醒播放：
系统支持播放预置音频，例如“前方有危险”“倒车请注意”“你吓到我了”等。音频文件存放于小车 `/home/jetson/car_audio/`，由车端音频播放模块拉起独立播放进程。

危险场景联动：
当 AI 检测发现高危隐患时，系统触发高可见度灯效和危险语音提示，例如红色闪烁灯光和“前方有危险”音频。

倒车避让联动：
当小车执行倒车、狭窄通道让行或避障动作时，系统触发黄色警示灯和倒车提示音，提醒现场人员保持距离。

巡航状态联动：
小车正常巡航时使用低亮度呼吸灯或固定巡航灯效，避免过度干扰现场人员，同时让车辆状态可识别。

多车角色联动：
多车编队或救援场景下，系统根据 leader、follower、responder 等角色触发不同灯效，便于现场区分车辆任务分工。

非阻塞控制要求：
灯光与音频播放不阻塞底盘运动控制。播放过程中仍允许紧急停止、人工接管和任务中断。

### 5.8 LLM 自然语言巡检助手功能

LLM 自然语言巡检助手用于降低巡检系统操作门槛。用户通过自然语言表达巡检意图，系统将其转换为可审计、可确认、受限制的任务计划。

自然语言任务输入：
App 提供聊天式助手入口，用户可输入“查询小车状态”“急停所有车辆”“让 robot_001 核验某区域车牌”“生成今日告警摘要”等自然语言指令。

任务计划生成：
后端根据用户输入生成任务计划，返回计划编号、步骤列表、目标车辆、安全级别、必要参数和安全提示。真实 LLM 未配置时，系统使用规则兜底模式完成基础任务。

工具白名单管理：
LLM 调用的能力来自后端工具白名单，包括车队总览、车辆列表、就绪检查、安全停止、短距离前进、通道巡航、通道避让、危险绕行、编队、救援搜索、车牌核验和护送返航等。

运动任务确认：
涉及小车运动的任务必须由用户确认后才能执行。未确认前，系统只展示计划，不下发真实运动指令。

就绪状态校验：
执行运动类任务前，后端检查目标小车是否在线、是否 ready、是否满足执行条件。不满足条件时，系统拒绝执行并返回原因。

执行结果反馈：
任务执行后，系统展示每个步骤的执行状态、后端接口返回、失败原因和安全提示。执行结果以真实接口返回为准，不由 LLM 猜测。

密钥安全管理：
LLM API Key 配置在后端环境变量中，App 不保存或展示 API Key。App 只查看 LLM 是否已配置、当前模型和规划模式。

### 5.9 多车协同巡检功能

多车协同功能用于提升大面积地下空间的覆盖能力。系统支持多车状态聚合、批量命令、编队巡检、狭窄通道协同、危险绕行、故障救援和护送返航。

多车状态聚合：
系统展示多台小车的在线状态、工作模式、电量、网络延迟、地图位置、最近心跳、编队角色和最近 ACK 状态。

批量命令下发：
系统支持向多台车批量下发同一命令，并为每台车生成独立 `command_id`。每台车的发布状态、ACK 状态和错误信息单独记录。

车队就绪检查：
在执行批量运动、编队或救援任务前，系统检查目标车辆是否在线、是否处于可执行状态，并返回每台车的就绪原因。

编队巡检：
系统支持创建编队任务，分配 leader、follower、队形类型、槽位和间距。编队成员在状态上报中携带编队 ID、角色和槽位信息。

狭窄通道协同：
在地下通道等狭窄区域，多车不同时抢占通道。系统支持低速分批通行、起步间隔控制和让行策略。

危险区域绕行：
当某台车发现积水、异物或其他危险点时，系统向相关车辆下发绕行、停止或避让命令，降低多车重复进入危险区域的风险。

故障救援：
当某台车故障或卡滞时，系统允许指定另一台车作为 responder，执行接近、搜索或辅助救援动作，并记录救援事件 ID。

护送返航：
当小车低电量、故障或任务异常时，系统支持另一台车护送其返回维修区或安全区域，并记录护送任务 ID、目标车和护送车。

### 5.10 停车定位、车牌核验与找车引导功能

停车定位、车牌核验与找车引导功能用于地下停车场巡检和车主服务场景。系统将车辆停放位置、车牌信息、地图坐标和用户手机绑定起来；当用户需要找车时，系统根据车牌在地图中定位车辆，并由巡检小车引导用户前往目标车位。

停车自动定位绑定：
用户停车后，系统记录车辆所在区域、车位编号、地图坐标、入场时间和车牌信息，并将该停车记录绑定到用户手机端。用户在 App 中能够查看自己的车辆停放位置和所在地图区域。

车牌识别与核验：
巡检小车在停车场巡检过程中识别并核验车牌，将车牌号、识别置信度、目标框、来源相机、所在区域和地图坐标写入记录。系统通过车牌信息将车辆身份与停车位置关联，形成可查询的车辆位置索引。

手机端找车请求：
用户在 App 中输入车牌号或选择已绑定车辆后，系统在地图中检索对应车辆位置，返回目标车位、当前位置到目标车辆的路径信息和引导状态。

地图定位与路线生成：
系统基于 SLAM 地图和停车位坐标，将用户当前位置、小车当前位置和目标车辆位置统一映射到地图坐标系中，并生成前往目标车辆的导航目标序列。

小车引导找车：
巡检小车接收找车任务后，前往用户附近或约定会合点，通过灯光和语音提示确认引导开始，再按地图路线低速引导用户前往目标车辆所在车位。

引导过程提示：
引导过程中，App 实时显示目标车辆位置、小车位置、剩余距离和当前引导状态；小车通过灯光提示行进状态，在转弯、到达和异常中断时播放对应语音提醒。

异常处理：
如果目标车辆位置不可达、车牌未匹配、路径被障碍物阻断或用户取消找车，系统停止引导任务，记录失败原因，并提示用户重新选择车辆或请求人工协助。

车牌核验任务：
系统支持输入车牌号、目标区域、相机编号和识别置信度等信息，生成车牌核验任务。

低速扫描核验：
小车执行车牌核验时，采用低速旋转或短距离移动方式扫描目标区域，避免影响停车场车辆和人员安全。

识别结果记录：
系统记录车牌号、识别置信度、目标框、区域、来源相机、核验车辆和任务状态，便于查询和复核。

LLM 触发核验：
用户通过 LLM 助手发起车牌核验任务，例如“让 robot_001 核验 B2 消防通道的沪A12345”。LLM 只生成任务计划，实际执行仍需后端安全校验和用户确认。

## 6. 数据管理需求

巡检价值不仅在于当下发现问题，更在于长期积累数据、支撑决策。系统采用 MySQL 作为持久化存储，数据范围包括：

- 用户与角色：支撑登录、权限管理和操作边界。
- 机器人档案：记录机器人编号、状态、模式、电量、网络、地图、最近心跳等。
- 告警记录：记录告警编号、类型、等级、状态、图片路径、位置、时间、处理备注。
- 检测记录：记录模型输出、检测类别、置信度、相机、图像来源和检测耗时。
- 操作日志：记录人为操作、控制命令、告警处理和关键配置变更。
- 多车命令：记录 `command_id`、目标车、命令内容、发布状态、ACK 和错误。
- 编队/救援/护送任务：记录多车协同任务 ID、角色、槽位、目标和执行结果。

系统支持统计分析，例如“某条通道本月出现过几次积水”“哪个区域异物告警最多”“哪台车离线频率最高”“某次高危告警由谁处理”。

## 7. 非功能性需求

### 7.1 实时性

- App TCP 遥控指令下发至小车响应延迟目标低于 200 ms。
- 小车位置、心跳与任务状态通过 MQTT 推送的端到端延迟目标低于 300 ms。
- 异常告警从触发到推送至 App 的延迟目标低于 500 ms。
- YOLO 检测模型经 TensorRT 或 Jetson GPU 加速后，目标推理帧率不低于 15 FPS。

### 7.2 稳定性

- 小车支持连续 2 小时自主巡检，不出现关键节点崩溃或控制链路失效。
- 网络中断恢复后，ROS、MQTT、后端服务自动重连。
- 对同一地点、同类异常的短时间重复触发进行去重，避免告警风暴。
- App 具备必要的异常提示和离线状态展示，不能静默失败。

### 7.3 安全性

- 运动类接口需限制速度、持续时间和执行条件。
- LLM 不能直接下发任意速度指令，只能调用后端白名单工具。
- 紧急停止优先级最高，能够覆盖巡检、导航、灯光秀和多车任务。
- 用户登录使用 JWT，关键操作需记录身份和时间。

### 7.4 可演示性

- App 功能可视化完整，支持一键进入地图、控制、告警、助手、设置等页面。
- 算法优化保留原厂对照代码、量化指标和演示视频。
- 每个核心场景能用明确链路验证，而不是只展示界面。
- 场景演示建议优先采用“手动遥控 -> 地图导航 -> 摄像头预览 -> AI 告警 -> 声光联动 -> LLM 辅助 -> 多车协同”的顺序。

### 7.5 部署便捷性

- 提供 Docker/脚本方式启动 ROS 容器、web_api、ros_bridge、ai_service、tcp_car_bridge 和 MQTT。
- 小车上电后核心服务支持自启动或一键启动。
- 车端项目路径统一为 `/home/jetson/Project/car10th`，容器内路径统一为 `/root/car10th`。
- 自动化部署采用 releases/current 结构，异常时可快速回滚到稳定版本。

### 7.6 持续迭代能力

- AI 检测模块支持新增类别，且不重构告警业务主流程。
- 声光联动策略已经从手动按钮抽象为事件驱动规则，支持不同巡检事件触发不同反馈。
- 多车协同接口支持更多车辆加入，不局限于两台演示车。
- LLM 工具白名单支持持续增加新工具，每个工具都定义安全级别、必填参数、确认要求和 readiness 检查。

## 8. 运维与部署需求

小车长期运行在无人值守的地下空间，系统需降低现场维护成本，具备自动化部署和快速恢复能力。

**部署要求**

- GitHub 主分支更新后，可通过 self-hosted runner 自动部署到小车端。
- 小车端采用 `releases/current` 目录结构管理版本。
- 核心服务由 systemd 或启动脚本守护，设备重启后可自动恢复。
- 部署路径为 `/home/jetson/Project/car10th`，容器内挂载为 `/root/car10th`。

**运行检查**

- `docker ps` 确认 `ros_x3_fixed` 运行。
- `curl http://127.0.0.1:8000/health` 检查 web_api。
- `curl http://127.0.0.1:8001/health` 检查 ros_bridge。
- `curl http://127.0.0.1:8002/health` 检查 ai_service。
- `ss -lntp | grep ':6001'` 检查 TCP 控制桥。
- App 读不到数据时，优先按 `docs/robot_daily_startup.md` 检查服务启动顺序。
